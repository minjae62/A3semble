# 테스트 시나리오 3 — 실제 API 기반 폐기량 절감 재측정

> 대상: `fridge-project 2` (레시피 추천 API가 실제 구현된 새 코드)
> 측정일: 2026-06-06 · 데이터: 라이브 백엔드(cloudtype)에서 추출한 `sim-data/`
> **측정 조건: 매일 2회 요리 · 주 1회 장보기 · 30일 · 유통기한 3주(21일) 이상 장기보관 재료 제외**

## 1. 무엇이 바뀌었나 (이전 vs 지금)

이전 `fridge-project`에서는 레시피 추천 API(F-03)가 "예정" 상태라 실제 레시피 DB가 없어
시나리오 3을 합성 fixtures(가짜 레시피 1,500종)로 **추정**했다.
지금 `fridge-project 2`는 레시피 API가 실제로 동작한다:

| 엔드포인트 | 역할 |
| --- | --- |
| `GET /api/v1/recipes?limit=&min_match_rate=` | 백엔드가 **α-score(`score`)·`rank` 직접 계산** |
| `GET /api/v1/recipes/{id}` | 단계 포함 상세 |
| `POST /api/v1/recipes/{id}/complete` | **FIFO 재고 자동 차감**, lot별 차감량 반환 |

## 2. 백엔드 로직 검증 (설계서 ↔ 구현) — `verify.mjs`

| 항목 | 결과 |
| --- | --- |
| 식재료 score = `risk × qty / ((D-day+1)² + 1)` | **45 / 45 일치** |
| 레시피 score = Σ(보유 매칭 재료 score) | **50 / 50 일치** |
| 랭킹 = score 내림차순 | 일치 |

**발견:** 설계서 원식은 `α×qty/((D-day)²+1)`인데 구현은 분모 D-day를 **+1 시프트**한 `((D-day+1)²+1)`을 쓴다
(D-day=0 점수 폭발 방지용 보수적 변형, 랭킹엔 영향 거의 없음). 설계서 수식만 갱신하면 문서-구현이 정확히 일치한다.

## 3. 레시피 DB 규모에 대한 정정

`/api/v1/recipes`는 **현재 재고와 겹치는 레시피만** 추려 주는 추천 엔드포인트다.
이 계정 재고(45건) 기준 추천 가능 레시피 = **50종**(모두 재고와 1개 이상 겹침, 겹침 0은 응답 없음).
따라서 **50 = "현재 재고로 추천 가능한 수"**이지 DB 전체 수가 아닐 수 있다. DB 전체 수는 라이브 재조회 필요.

## 4. 시뮬레이션 설계 — `sim.mjs`

- **데이터:** 실제 레시피 50종, 식재료 마스터 427종(risk·shelf_days).
- **이번 조건:** **매일 2회 요리** · 장보기 **주 1회**(7일마다) · **30일** · **유통기한 3주(21일) 이상 제외**(71종 제외, 신선재료 84종 대상).
- **방법:** 1,000명 가상 사용자. 같은 사용자에게 **동일 입력**(초기 재고·장보기·끼니별 무작위 추첨)을 주고
  레시피 **선택 전략만** 분기(paired) → 전략 효과만 분리. 하루 2끼 각각 추천/무작위로 골라 FIFO 차감.
  - α-score(우리 시스템): `recipeScore = Σ 보유 매칭 재료 score` 최댓값(백엔드와 동일).
  - 무작위: 매칭 레시피 중 무작위.

## 5. 결과 (1,000명 · 30일 · 매일 2회 · 주1회 · 신선재료만)

| 지표 | 무작위 | α-score | 차이 |
| --- | --- | --- | --- |
| 1인 평균 폐기량 | 18,232 g | 13,764 g | **−24.5%** |
| 폐기율(투입 대비) | 56.5% | 42.7% | −13.9%p |
| 개선된 사용자 비율 | — | — | **97.0%** |

**해석:** 요리를 하루 2회로 늘리니 소비량이 커져 α-추천이 임박 재료를 태울 기회가 많아졌고,
절감률이 1회(10.3%) → **2회 24.5%**로 크게 올랐다. 폐기율도 66%→43%로 떨어졌다.

> 참고: 폐기율 절대값은 그램 환산·구매량 가정에 민감하다. 전략 간 **상대 절감률(24.5%)** 이 더 견고한 지표다.

## 6. 민감도 분석

| 레버 | 범위 | 절감률 |
| --- | --- | --- |
| 하루 요리 횟수 | 1 → 4회 | 14.4% → 24.0% → 31.8% → **38.9%** |
| 레시피 수 | 10 → 50종 | 4.1% → **24.0%** |
| 보유 품목 수 | 12 → 36개 | 29.1% → 19.4% (절대 폐기량은↑) |

요리 횟수·레시피 수가 가장 큰 레버. 둘 다 클수록 α-추천 이득이 커진다.

## 7. 한계 / 다음 단계

- 추천 응답 50종 = 현재 재고로 추천 가능한 수(§3). DB 전체 수 확인은 라이브 재조회 필요.
- `complete`(FIFO) 응답 구조는 코드로 확인, 차감 규칙은 로컬 재현. 규칙 일치는 score 검증으로 담보.
- 그램 환산표·구매량은 가정값. 실제 1회 제공량/구매량으로 교체 시 절대 폐기율이 정밀해진다.

## 8. 재현 방법

```bash
node pull_recipe_data.mjs      # (네트워크 PC) 실제 데이터 추출 -> sim-data/
node scenario3/verify.mjs      # 백엔드 score 전수 대조
node scenario3/sim.mjs         # 폐기량 재측정 + 민감도 -> sim_results.json
```
조건 변경은 `sim.mjs` 상단 `CFG`: `cooksPerDay`(하루 요리 횟수), `restockEvery`(장보기 주기), `days`, `excludeShelfDaysGte`(장기보관 제외 기준일).
